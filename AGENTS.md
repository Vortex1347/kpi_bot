## Documentation Sync Rule (mandatory)

После каждого изменения кода (fix/refactor/feature) обязательно обновлять релевантные документы в этом же change set.

### Что обновлять

- Если изменились API-контракты, DTO, статусы, auth/cookie flow:
  - `docs/openapi.yaml`
  - при архитектурном решении: `docs/adr/*`
- Если изменились архитектурные границы, слои, feature decomposition:
  - `ARCHITECTURE.md`
  - `docs/DECOMPOSITION.md`
  - `PROJECT.md`
- Если изменился запуск/настройка/deploy/окружение:
  - `README.md`
  - `START_HERE.md`
  - `docs/DEPLOYMENT_DOMAINS.md`
  - `backend/.env.example` и/или `frontend/.env*.example`
- Если изменились тестовые сценарии/команды/покрытие:
  - `TESTING.md`
  - при необходимости `backend/test/README.md`

### Правило ответа

В конце каждого выполненного запроса явно перечислять, какие docs были обновлены.

## MCP Usage Policy (token optimization)

Использовать MCP-инструменты по приоритету, чтобы снижать расход токенов и не грузить лишний контекст.

### Активно используемые MCP плагины

- `serena`:
  - приоритет №1 для работы с кодом в репозитории;
  - сначала использовать символические/точечные операции (`find_symbol`, `get_symbols_overview`, `search_for_pattern`), а не читать целые файлы;
  - читать только необходимые файлы/фрагменты.
- `context7`:
  - использовать только для внешней документации библиотек/фреймворков;
  - сначала `resolve-library-id`, затем точечный `query-docs`;
  - не запрашивать общие/широкие справки, если ответ уже есть в локальном коде.
- `playwright`:
  - использовать для e2e smoke, UI-проверок и скриншотов;
  - предпочитать `browser_snapshot`/точечные действия, а не длинные последовательности без необходимости.

### Практика экономии токенов

- Сначала локальный код и локальные docs, потом внешние источники.
- Не дублировать один и тот же контент в ответе.
- Не читать большие документы полностью, если достаточно конкретного раздела/поиска.
- В финальном ответе давать краткое резюме + ссылки на измененные файлы.
